## 1. Background & Problem Statement

In silicon design verification (DV), a **Subsystem Testplan** maps every
functional requirement (Feature / FID / PRD item) of a sub-system to one or
more verification entries — testcases, functional coverpoints, protocol
checkers, and milestone tags.  Today this is written manually by senior DV
engineers and takes days to weeks.

The skill should automate this end-to-end: ingest the design specification
documents, extract the DV-relevant information, and generate a professional,
team-ready Excel testplan.

| # | Input | Description | If Missing |
|---|-------|-------------|------------|

## 3. Output & Format

### Primary Output: `testplan.xlsx`

A multi-sheet Excel workbook:

**Sheet 1 — Summary**
Auto-generated statistics table:
- Total testplan rows (directed / random / coverpoint-only / negative / corner / stress)
- Milestone breakdown: DV-I / DV-C / DV-F counts
- Covergroup, coverpoint, cross-coverpoint counts
- Checker counts (Procedural / Assertion / Both)
- Features covered vs total features in spec

**Sheet 2 — Testplan** (11 columns)

| Col | Name | Content |
|-----|------|---------|
| 1 | PRD Number | PRD reference if available; blank if none |
| 2 | Feature | Top-level feature name + FID tag |
| 3 | Sub-Feature | Sub-division; prefix `[NEG]` / `[STRESS]` / `[CORNER]` where applicable |
| 4 | Description | 1–2 sentence feature description |
| 5 | Verification Type | One or more of: `Testcase(Directed)`, `Testcase(Random)`, `Coverpoint`, `Checker` — newline-separated |
| 6 | Testcase Description | DUT Config / Stimulus / Expected Behavior / Checks / Pass Criteria / Notes |
| 7 | Coverpoint Description | Syntactically correct SystemVerilog covergroup/coverpoint code |
| 8 | Checker ID | Format: `CHK_<IP>_<FEATURE>_<TYPE>_<NNN>` — globally unique |
| 9 | Checker Type | `Procedural` / `Assertion` / `Both` |
| 10 | Checker Implementation | `scoreboard` / `monitor` / `interface` / `testcase` |
| 11 | Milestone | `DV-I` / `DV-C` / `DV-F` |

### Intermediate Outputs (written to `.dv_ir/` in OUTPUT_DIR)

These JSON files are reusable by downstream skills — do **not** delete them
after the run:

| File | Contents | Reused by |
|------|----------|-----------|
| `spec_ir.json` | Extracted DUT overview, features, registers, interfaces, clocks, resets, interrupts, memory map, protocols | Any skill that needs spec data |
| `fid_prd_map.json` | FID list, PRD-FID mapping, inferred FIDs, unmapped items | Testplan generator, verstrat generator |
| `testplan_ir.json` | All testplan rows as JSON array — 11-column schema | XLSX formatter, verstrat generator |
| `benchmark_report.md` | Benchmark scores and gap analysis (only if `--benchmark*` flag used) | Reporting |

These intermediate files enable downstream skills (e.g. interface VIP
development, simulation setup, scoreboard generation) to reuse the extracted
spec data and the parsed testplan rows **without re-reading the spec**.
Examples of reuse:
- `spec_ir.json → interfaces[]` used by the interface VIP skill
- `spec_ir.json → registers[]` used by the RAL generation skill
- Common scripts (xls reader, JSON writer) should live in `skills/common/scripts/`
  so all skills share them rather than duplicating

## 4. Skill Structure

Design the skill to be **modular, reusable, generic, and token-efficient**:

### 4.1 Sub-Skills / Sub-Steps

Break the skill into the following discrete sub-steps, each with a clear input,
output, and bounded token budget:

| Sub-step | Input | Output | Notes |
|----------|-------|--------|-------|
| `spec-ingester` | Raw spec files (PDF/DOCX/XLSX) | `spec_ir.json` | Two-pass: structure scan → targeted section extraction |
| `prd-fid-extractor` | `spec_ir.json` + optional PRD file | `fid_prd_map.json` | Three modes: separate file / embedded / inferred |
| `testplan-generator` | `spec_ir.json` + `fid_prd_map.json` | `testplan_ir.json` | Batched by feature group to stay within context |
| `xlsx-formatter` | `testplan_ir.json` | `testplan.xlsx` | Uses `skills/common/scripts/generate_testplan_excel.py` |
| `benchmarker` | `testplan_ir.json` + existing docs | `benchmark_report.md` | Only runs if `--benchmark*` flag passed |

### 4.2 Token & Context Optimisation

- Parse specs in **two passes**: first extract structure/TOC/headings, then
  extract content only from DV-relevant sections — never pass the full raw
  spec text to the LLM.
- Pass only compact IR JSON between sub-steps — never raw spec text.
- Batch testplan generation: **10–15 FIDs per LLM call** with carry-forward
  registry for checker IDs and testcase names (uniqueness across batches).
- Cache invalidation: if `spec_ir.json` already exists and the spec file
  has not changed (compare `mtime`), skip re-ingestion.
- If context approaches 100%, write the current batch output to file and
  start a fresh context — never let context saturation degrade output quality.

### 4.3 Reusability

- All Python helper scripts go in `skills/common/scripts/` — not inside
  this skill's directory.
- The XLSX generation script, JSON writer, and document reader should be
  shared with existing skills (`dv-testplan`, `dv-spec-parse`, etc.).
- The skill should be **parameterisable by project type** (IP / SS / SoC)
  so the same core logic can be extended later without a rewrite.

## 5. Benchmarking

The skill must support three benchmarking modes, activated by flags:

### 5.1 `--benchmark` — vs Existing Human Testplan
Compare generated `testplan.xlsx` against a human-written testplan.

Metrics (score each 0–1):
- **Feature Recall** = |generated ∩ existing features| / |existing features|
- **Feature Precision** = |generated ∩ existing features| / |generated features|
- **TC Count Ratio** = generated TC count / existing TC count
- **Coverpoint Count Ratio** = generated / existing
- **Checker Count Ratio** = generated / existing
- **Missing Features** = list of features in existing but not generated
- **New Features Found** = list of features generated but not in existing (genuine gaps)
- **Overall Score** = weighted: features×0.30, testcases×0.30, coverage×0.25, checkers×0.15

### 5.2 `--benchmark-mda` — vs MDA Agent Output
Same metrics as above but the baseline is output from MDA (another AI agent).
Flag: `--benchmark-mda <mda_testplan.xlsx>`

### 5.3 `--benchmark-prev` — vs Previous Skill Version
Compare current output against output from a previous version of this skill.
Flag: `--benchmark-prev <prev_testplan.xlsx>`
Useful to confirm that a skill update improved quality and did not regress.

All benchmark results are written to `benchmark_report.md` in OUTPUT_DIR.

## 6. Good Practices (Mandatory — encode in SKILL.md)

The skill **must** enforce these practices in its instructions to the agent:

### 6.1 Stop on Error — Never Silently Substitute
- If a required input file is missing or unreadable: **stop and tell the user**
  clearly.  Do not attempt to proceed with a substitute or guess.
- If a Python script fails: print the full error, then **stop**.  Do not try
  an alternative approach that might produce lower-quality output.
- If a required Python package is missing: run the environment-check script
  and auto-install. If install fails: stop and report.

### 6.2 Monitor Processing Quality
- After ingesting each spec file, print a one-line summary:
  `[spec-ingester] <filename>: <N> features, <N> registers, <N> interfaces extracted`
- After each testplan generation batch, print:
  `[testplan-gen] Batch <N>/<total>: <N> rows generated, cumulative: <N> rows`
- **Context warning**: if context usage exceeds 80%, print a warning and
  checkpoint: flush current output to file and log the checkpoint.
- After Excel generation, print the file path and row count for confirmation.
- Any step that produces fewer results than expected (e.g. 0 features
  extracted from a non-empty spec) should trigger a warning — not silently
  produce an empty output.

### 6.3 Python Virtual Environment
- Always use a local virtual environment for Python packages.
- Check for `venv/` in `skills/common/`; create it if absent.
- All `pip install` commands must target this venv — never install globally.
- The environment check script (`check_environment.py`) must activate the
  venv before running any other script.

### 6.4 Clean Agent Workspace
- Do **not** leave intermediate files (raw spec text dumps, temp JSONs,
  debug logs) in the agent workspace or OUTPUT_DIR after a successful run.
- Only the final deliverables remain: `testplan.xlsx`, `benchmark_report.md`,
  and the `.dv_ir/` IR files.
- Temp files go to `/tmp/<project>_dv_ss_testplan_*` and are cleaned up
  at the end of a successful run.

### 6.5 Common Infrastructure — Reduce AI Token Load
- Register a shared **document format specification** in
  `skills/common/doc_formats.md` so the ingester doesn't re-discover format
  quirks every run.
- Shared scripts in `skills/common/scripts/` must be documented in a
  `SCRIPTS.md` index file so any skill can discover and reuse them.
- For large specs: extract text to a pre-processed plain-text cache file
  (`.dv_ir/spec_text_cache.txt`) before LLM calls — avoids re-parsing PDF
  bytes repeatedly.

### 6.6 Context Reset Between Tasks
- Between major sub-steps (ingestion → FID extraction → testplan generation
  → formatting), use a **compact handoff**: pass only the IR JSON filename,
  not the full IR content, to the next sub-step prompt.
- Start each sub-step with a concise context header (5–10 lines) rather than
  re-explaining the full problem.
- If the agent runs a long session, checkpoint at the end of each sub-step
  so work is not lost if the session is interrupted.

## 8. Iteration & Improvement Loop

After the first version of the skill is created:

1. **Run the skill** on 2–3 real subsystem specs.
2. **Run benchmarking** (`--benchmark`) against existing human-written
   testplans for those projects.
3. **Review the benchmark report** — identify which dimension scores lowest
   (feature recall? testcase coverage? coverpoints?).
4. **Review generated testplan vs existing** side-by-side — note specific
   missing entries, incorrect checker IDs, weak testcase descriptions, etc.
5. **Give structured feedback** to the Skill Creator:
   - What was missing
   - What was wrong
   - What good patterns to preserve
6. **Update the skill** via Skill Creator — increment the version in SKILL.md.
7. **Re-run benchmarks** to confirm scores improved and no regression.
8. Repeat until overall testplan score ≥ 0.85.
